# 避坑记录

## 使用规则
- 只记录已确认不通、重复出错或明显浪费时间和 token 的路径。
- 每条尽量简短，优先写“错误路径 -> 为什么不通 -> 下次做法”。
- 新会话接手时先快速浏览本文件，再开始执行。
- 同类问题已记录后，后续不要重复试同一条无效路径，除非条件已变化。
- 同一路径、方法、命令、工具调用方式或环境假设连续两次失败时，必须记录。
- 即使只失败一次，但代价高、误导强、容易重复踩，也应直接记录。
- 不要用兜底策略来进行补充修改；如果某条路径本身不成立，应记录失败原因并回到根因处理。

## 当前有效避坑
- 在没有实际项目文件前，不进行代码级分析、测试、依赖安装或 git 操作推断。
- 遇到“目录空 / No-Git”时，优先判断是目录错了还是需要初始化，不要继续深挖不存在的上下文。
- GitHub 推送前先核对当前目录是否是仓库、是否已有远端，再决定提交和推送动作。
- 遇到 `/dev/tty` 和 `failed to execute prompt script` 时，优先怀疑 askpass/helper 链路，而不是仓库权限。
- 核对记忆迁移时，不要先假设目标端一定有单个 `memory.md`；先确认目标工具实际采用单文件还是目录结构。
- 跨设备接入 Codex 时，优先让共享文件落到本地磁盘，再用软链接接入；不要把 Codex 关键目录直接绑到不稳定的网络挂载。
- 处理问题时，不要因为主路径失败就叠加兜底补丁；应先确认失败原因，再决定是否切换方案。
