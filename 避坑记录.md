# 避坑记录

## 使用规则
- 只记录已确认不通、重复出错或明显浪费时间和 token 的路径。
- 每条尽量简短，优先写“错误路径 -> 为什么不通 -> 下次做法”。
- 新会话接手时先快速浏览本文件，再开始执行。
- 同类问题已记录后，后续不要重复试同一条无效路径，除非条件已变化。

## 条目模板
| 日期 | 场景 | 错误路径 | 为什么不通 | 下次优先做法 | 状态 |
|------|------|----------|------------|--------------|------|
| 2026-03-25 | 空目录接手 | 直接按有项目假设做 git/代码分析 | 当前目录为空且 `No-Git`，会导致无效检查和误判 | 先做启动检查，再决定是初始化模板还是切到真实项目目录 | active |
| 2026-03-25 | GitHub 推送 | 在未验证 `.git` 和远端前直接承诺可以推送 | 没有仓库和远端信息时无法提交或推送 | 先执行 `if (Test-Path .git) { git status --short; git remote -v } else { 'NO_GIT_REPO' }` | active |

## 当前有效避坑
- 在没有实际项目文件前，不进行代码级分析、测试、依赖安装或 git 操作推断。
- 遇到“目录空 / No-Git”时，优先判断是目录错了还是需要初始化，不要继续深挖不存在的上下文。
- GitHub 推送前先核对当前目录是否是仓库、是否已有远端，再决定提交和推送动作。
